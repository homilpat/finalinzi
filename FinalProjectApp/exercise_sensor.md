# 운동 코칭 정리 문서

작성 기준: `exercise.js` 코드 리뷰
작성 목적: Kotlin 센서 연동 작업에서 "언제, 어떤 동작을, 어떻게 판정해야 하는지" 파악하기 위함

\---

## 1\. 센서 이벤트 통신 규격

```javascript
// Kotlin → WebView
webView.evaluateJavascript("window.SensorBridge.onSensorEvent('{\\\\\\\\"action\\\\\\\\":\\\\\\\\"step\\\\\\\_right\\\\\\\\"}')", null)
```

* JSON 문자열 하나, `action` 필드만 사용
* 판정이 이루어지는 구간은 \*\*1단계(워밍업)\*\*와 **3단계(본운동/듀얼과제)** 뿐입니다. **2단계(신체강화)·4단계(쿨다운)는 센서 판정을 하지 않습니다.**
* 테스트 버튼(`triggerVirtualSensor`)도 실제 프로덕션 경로(`SensorBridge.onSensorEvent`)를 그대로 거치므로, 테스트로 확인되는 동작 = 실제 센서 연동 시 동작입니다.

## 2\. 액션 어휘 전체 목록 (Kotlin이 만들어내야 하는 값, 8종)

이제 모든 유형·모든 구간에서 **동의어 없이, 정확히 이 이름들만** 사용합니다.

|action 값|사용처 (유형·단계)|판정 방식 힌트|
|-|-|-|
|`stop`|A/B 1단계, A 3단계|선형가속도+자이로 크기가 임계값 이하로 일정시간 유지|
|`weight\_right`|A/B 1단계, C 1단계|중력벡터 우측 성분 + 최소 유지시간(hold)|
|`weight\_left`|A/B 1단계, C 1단계|중력벡터 좌측 성분 + 최소 유지시간(hold)|
|`step\_forward\_right`|A/B 1단계|전방 스텝 + 우측 부호|
|`step\_backward`|A/B 1단계|후방 스텝|
|`step\_right`|B 3단계|수직 충격 피크 + 우측 방향 부호|
|`step\_left`|B 3단계|수직 충격 피크 + 좌측 방향 부호|
|`knee\_extension`|C 3단계|신호 약함 — 캘리브레이션 파형 템플릿 비교 권장|
|`any\_reaction`|**D유형 전용, 1단계·3단계 공통**|가장 낮은 임계값, 정오답 없음 — D유형은 이 값 하나만 알면 됨|

> `step\_forward\_right`라는 이름이 살짝 특이한데(전방+우측이 합쳐진 이름), "오른발을 앞으로 한 발"이라는 특정 동작을 가리키는 것으로 보입니다. 뒤이은 `step\_backward`는 방향 표시가 없어 명명 규칙이 완전히 대칭적이진 않지만, 기능상 문제는 없으니 참고만 해주세요.

## 4\. 유형별 상세 명세

### A유형 — 1단계 (워밍업, BGM: `A\_1단계워밍업.mp3` 공용)

두 유형이 완전히 동일한 판정 로직을 공유합니다.

|순서|BGM 재생시간 구간|TTS/가사 큐|기대 action|
|-|-|-|-|
|1|53\~56초|"멈춰!"|`stop`|
|2|67\~73초|"오른쪽 엉덩이로 체중을 눌러요"|`weight\_right`|
|3|78\~81초|"왼쪽으로 누르세요"|`weight\_left`|
|4|116\~122초|"오른발 앞으로 한발"|`step\_forward\_right`|
|5|124\~125초|"이번엔 뒤로 한발"|`step\_backward`|

⚠️ **구간 5(124\~125초)는 여전히 1초 폭입니다.** 다른 구간은 3\~6초인데 이것만 유난히 좁습니다. 오탈자인지 의도인지 기획/오디오 담당자 확인을 권장하며, Kotlin 판정 주기가 느리면 이 구간을 통째로 놓칠 수 있습니다.

⚠️ **시간구간 자체는 아직 실측 검증되지 않았습니다.** `A\_1단계워밍업.mp3`를 직접 음향분석했을 때 뚜렷한 휴지 구간은 약 50.8초/53.6초 부근이었고, 코드의 53\~56초와는 근접하지만 나머지(67\~125초 구간들)는 검증 전입니다. STT로 가사 타임스탬프를 재확인하는 걸 권장합니다.

**2·4단계**: 센서 판정 없음.

### A유형 3단계 (본운동, 9분) — 카테고리 규칙 기반 정지 판정

무릎 높이 들며 제자리 걷기를 계속하는 중, 단어를 듣고 규칙에 맞으면 `stop` 판정.

|165\~185s|규칙2 안내 (반전: 동물→멈춤)|없음|
|185\~305s|2라운드 진행|동물 단어일 때만 `stop`|
|305\~325s|중간 호흡 확인|없음|

|340\~460s|3라운드 진행|과일 단어일 때만 `stop`|
|460\~475s|규칙4 안내 (재반전 + 고속)|없음|

## 5\. Kotlin 개발 시 참고할 설계 원칙 요약

1. **멀티클래스 분류가 아니라 이진 확인으로 접근하세요.** JS가 이미 매 순간 "지금 뭘 기대하는지" 알고 있으므로(`waitingSensorAction`, `phaseGameData.targetAction` 등), Kotlin은 원시 센서값이 그 기대 동작의 특징에 부합하는지만 확인하면 됩니다.
2. **중력 성분과 선형가속도를 분리하는 전처리가 모든 판정의 기반입니다.**

   * 중력 성분(저역통과) → `weight\_right/left`처럼 "기울인 자세 유지"를 보는 동작
   * 선형가속도(원시값 − 중력성분) → `step\\\\\\\_\\\\\\\*`처럼 "순간적 충격/피크"를 보는 동작
3. **캘리브레이션(1단계 진입 시 `startCalibration()` 호출 시점)에서 좌우축 매핑을 저장**해두고, 이후 모든 좌우 판정에 재사용하세요. 착용 방향에 따라 실제 좌우가 어느 센서축인지 달라질 수 있습니다.
4. **모든 액션 이름이 이제 동의어 없이 1:1로 고정**돼 있으니, 2장 표를 기준으로 정확한 문자열을 맞춰주세요. 특히 D유형은 `any\_reaction` 하나로 끝나므로 가장 구현이 간단한 유형입니다.
5. **중복 이벤트 방지(디바운스)가 JS 쪽에는 없습니다.** 한 번의 실제 동작에 대해 여러 이벤트를 쏘지 않도록 "동작 시작\~종료"를 하나의 이벤트로 묶어서 전송해주세요. (B유형처럼 시퀀스 인덱스를 카운트하는 로직은 중복 이벤트에 취약합니다.)
6. **`knee\_extension`은 간접 신호만 잡을 수 있습니다.** 허리 부착 폰으로는 무릎 동작을 직접 감지하지 못하므로, 정밀 판정이 어려우면 초기 MVP는 `any\_reaction` 수준(움직임 발생 여부만)으로 완화하는 것도 현실적 대안입니다.

임계값(가속도/자이로 크기, 유지시간 등)은 모두 출발점 추정치이며, 실제 어르신 대상 시범 데이터로 보정하는 과정이 필요합니다.
